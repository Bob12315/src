# 基于视觉的无人机跟踪系统总策划书

## 1. 项目总述

本项目是一个基于 YOLO 的无人机视觉跟踪系统，采用“机载极简 + 地面计算”的总体架构。

核心定义如下：

无人机机体上只保留飞控及必要通信模块，所有视觉感知能力和高层控制逻辑都部署在地面计算机上，通过图传链路和数传链路与飞行器连接，形成闭环跟踪系统。

一句话概括：

The drone only carries a flight controller, while all perception (YOLO) and control (ROS2) run on a ground computer via telemetry and video link.

从工程定位上看，这不是一个“全机载自治系统”，而是一个“地面算力主导的视觉跟踪系统”。其重点不在于把所有智能都塞进无人机，而在于通过合理分层，把飞行安全相关功能留在飞控，把视觉与任务决策放到地面侧实现。

## 2. 理解总结与总体策划判断

### 2.1 我对项目的理解

这个系统本质上是一个分布式闭环控制平台，职责划分非常清晰：

- 飞机端负责飞行稳定、状态估计、执行控制和通信收发。
- 地面端负责目标识别、目标跟踪、融合判断与控制决策。
- 通信链路负责把视频送到地面，把控制指令送回飞控。

也就是说，系统的“眼睛”和“脑”在地面端，系统的“平衡”和“肌肉”在飞控端。

### 2.2 该方案的价值

这个方案非常适合以下目标：

- 快速验证视觉算法
- 降低机载计算负担
- 充分利用地面 GPU 算力
- 便于调试、记录和复现问题
- 易于扩展室内、室外、降落、仿真等不同应用场景

### 2.3 该方案的边界

该架构的能力边界也非常明确：

- 图传会引入显著延迟
- 依赖通信链路稳定性
- 更适合低速、中低动态目标跟踪
- 不适合高机动、高速追逐类任务

因此，这个系统应当被定义为“面向工程可落地与快速迭代的地面计算视觉跟踪架构”，而不是“追求极限实时性的全机载高速自主飞行平台”。

## 3. 项目定义

### 3.1 项目类型

本项目是一个融合以下关键技术的无人机系统：

- YOLO 目标检测与视觉跟踪
- ROS2 节点化控制系统
- MAVLink 飞控通信接口
- ArduPilot 或 PX4 飞控生态
- 地面计算式部署架构

### 3.2 项目目标

在尽量简化机载系统的前提下，实现无人机基于视觉目标信息的自动跟随、对准和任务执行能力。

### 3.3 核心设计哲学

本项目遵循以下设计原则：

- 飞机端尽量简单、稳定、可靠
- 地面端负责智能算法和高层逻辑
- 控制链路中只传输结构化数据，不传大图像流
- 飞控继续负责低层闭环控制和状态估计
- 控制只使用最新有效数据，避免历史缓存拖慢系统

## 4. 系统总体架构

系统由三大部分组成：

- 飞机端
- 地面端
- 通信链路

### 4.1 飞机端

飞机端只包含最低必要模块：

- 飞控：ArduPilot 或 PX4
- 机载传感器：IMU、GPS、光流、激光测距等，统一接入飞控
- 数传模块
- 图传模块

飞机端职责：

- 姿态解算
- EKF 状态估计
- 电机与姿态闭环控制
- 飞行模式管理
- 遥测发送与指令接收
- 视频发送

飞机端明确不负责：

- 不运行 YOLO
- 不运行 ROS2 主控制逻辑
- 不直接承担高层目标识别与任务决策

### 4.2 地面端

地面计算机是整个系统的感知与决策核心。

地面端运行：

- YOLO：负责视觉检测与跟踪
- ROS2：负责状态接入、融合、控制与执行管理

地面端职责：

- 接收图传视频
- 进行目标检测与目标定位
- 接收飞控遥测状态
- 融合视觉信息与飞行状态
- 计算目标跟踪控制量
- 通过 MAVLink 向飞控发送控制指令

### 4.3 通信链路

通信链路是整个系统能否闭环工作的核心基础。

系统中存在两条链路：

- 视频链路：单向，从无人机到地面端
- 控制链路：双向，飞控与地面端之间的遥测与指令通信

视频链路：

Drone -> Camera -> Video TX -> Video RX -> OTG/采集输入 -> YOLO

控制链路：

Flight Controller <-> Telemetry <-> Ground Computer <-> ROS2 <-> MAVLink Command

## 5. 总体架构图

```text
              【Drone】

     Camera ───→ Video TX ─────────────┐
                                       │
     Flight Controller ←→ Telemetry ←──┼──→ Ground Computer
                                       │
                                       ↓

              【Ground Computer】

     Video RX → YOLO → /perception/target

     Telemetry → /drone/state

     Fusion → Control → /control/cmd

     /control/cmd → Telemetry → Drone 
```

## 6. 核心数据流

系统数据流逻辑如下：

1. 图传视频送达地面计算机
2. YOLO 处理当前视频帧并输出目标信息
3. 遥测链路提供当前无人机状态
4. 融合层将视觉目标与飞行状态组合
5. 控制层计算速度、航向或姿态相关控制指令
6. 执行层通过 MAVLink 向飞控下发指令
7. 飞控完成底层闭环控制与执行

可简化为四条主线：

- Video -> YOLO -> Target
- Telemetry -> Drone State
- Target + State -> Control
- Control -> Flight Controller

## 7. 系统分层设计

为了保证后续实现清晰、可维护、易扩展，系统采用六层设计。

### 7.1 第一层：输入层 Input

输入层负责地面系统外部输入的统一接入。

#### 视频输入

来源可以包括：

- `/dev/video*`
- RTSP
- USB 采集卡
- OTG 接入的视频接收器

设计要求：

- 尽量减少视频数据拷贝
- 提供稳定时间戳
- 只保留最新帧，避免积压

#### 遥测输入

协议：

- MAVLink

物理接口：

- 串口
- USB
- 无线数传模块

设计要求：

- 稳定获取飞行状态
- 具有心跳检测能力
- 识别飞行模式、解锁状态和链路状态

### 7.2 第二层：感知层 Perception

感知层由独立 YOLO 进程组成。

职责包括：

- 目标检测
- 目标定位
- 目标跟踪
- 检测置信度输出

典型输出格式：

```json
{
  "target_valid": true,
  "cx": 320,
  "cy": 240,
  "ex": 0.05,
  "ey": -0.02,
  "confidence": 0.9
}
```

字段含义：

- `target_valid`：当前是否存在有效目标
- `cx`、`cy`：目标中心像素坐标
- `ex`、`ey`：相对画面中心的归一化误差
- `confidence`：检测置信度

设计说明：

- YOLO 是实时感知模块
- 感知输出只负责给出结构化目标结果
- 控制链路不依赖图像本体，只依赖目标结果

### 7.3 第三层：状态层 State

状态层主要来自飞控遥测。

典型输出：

```json
{
  "roll": 0.01,
  "pitch": -0.02,
  "yaw": 1.57,
  "altitude": 3.0,
  "mode": "GUIDED"
}
```

飞控负责：

- IMU 融合
- GPS 融合
- 光流与激光测距融合
- EKF 状态估计
- 模式切换与飞行安全机制

这一层非常关键，因为它明确了一个原则：

ROS2 不是飞控替代品，ROS2 只消费飞控已经整理好的飞行状态。

### 7.4 第四层：融合层 Fusion

融合层输入包括：

- 视觉目标信息
- 飞行状态信息

典型输出：

```json
{
  "target_valid": true,
  "ex": 0.05,
  "ey": -0.02,
  "tracking": true
}
```

融合层职责：

- 对齐目标信息与飞行状态
- 检查数据时效性
- 剔除过期目标
- 判定是否进入跟踪状态
- 输出供控制器直接使用的误差信号

### 7.5 第五层：控制层 Control

控制层负责把视觉误差转换成飞行控制指令。

典型输出：

```json
{
  "yaw_rate": 0.3,
  "vx": 0.5,
  "vy": 0.0,
  "vz": 0.0
}
```

控制层职责：

- 根据水平误差生成偏航角速度
- 根据目标位置或尺度生成前向、侧向速度
- 进行限幅与增益控制
- 在目标丢失时执行安全策略
- 固定频率运行，一般建议 20 至 50 Hz

控制设计原则：

- 控制器永远基于“最新有效融合结果”工作
- 不等待图像和状态完全逐帧严格对齐
- 重点是降低控制延迟，而不是追求理论上最完美的同步

### 7.6 第六层：执行层 Execution

执行层通过 MAVLink 将控制量发送给飞控。

职责包括：

- 将 ROS2 控制输出映射为飞控可接受的 MAVLink 指令
- 发送速度、偏航或姿态设定值
- 对控制发送频率进行约束
- 在数据超时或异常时及时停止控制输出

最终闭环由飞控完成，飞控负责执行器层面的稳定控制。

## 8. ROS2 节点设计

建议 ROS2 节点如下：

### 8.1 `perception_bridge`

功能：

- 接收 YOLO 输出
- 统一目标消息格式
- 发布目标检测结果

输入：

- YOLO 进程输出

输出：

- `/perception/target`

### 8.2 `telemetry_node`

功能：

- 连接飞控 MAVLink
- 读取遥测状态
- 发布飞行状态

输出建议：

- `/drone/state`
- `/link/status`
- `/system/heartbeat`

### 8.3 `fusion_node`

功能：

- 融合目标信息与飞行状态
- 检查数据新鲜度
- 输出跟踪状态

输入：

- `/perception/target`
- `/drone/state`

输出：

- `/mission/state`
- `/tracking/state`

### 8.4 `controller_node`

功能：

- 计算跟踪控制量
- 实现增益、限幅、目标丢失逻辑

输入：

- `/tracking/state`

输出：

- `/control/cmd`

### 8.5 `executor_node`

功能：

- 将控制命令通过 MAVLink 下发到飞控

输入：

- `/control/cmd`

输出：

- 飞控控制指令

## 9. Topic 设计

核心 Topic 建议如下：

- `/perception/target`
- `/drone/state`
- `/mission/state`
- `/control/cmd`

含义说明：

- `/perception/target`：视觉模块输出的结构化目标结果
- `/drone/state`：飞控输出的飞行状态
- `/mission/state`：当前任务状态，如待机、跟踪、丢失、降落等
- `/control/cmd`：控制器输出给执行层的高层控制指令

后续可扩展 Topic：

- `/tracking/state`
- `/system/health`
- `/link/status`
- `/landing/target`
- `/debug/perception`

## 10. 消息接口建议

虽然前期可以用 JSON 便于联调，但正式工程建议定义 ROS2 自定义消息。

建议消息类型：

- `TargetInfo.msg`
- `DroneState.msg`
- `TrackingState.msg`
- `ControlCmd.msg`
- `MissionState.msg`

### 10.1 `TargetInfo.msg` 建议字段

- `bool target_valid`
- `float32 cx`
- `float32 cy`
- `float32 ex`
- `float32 ey`
- `float32 confidence`
- `builtin_interfaces/Time stamp`

### 10.2 `DroneState.msg` 建议字段

- `float32 roll`
- `float32 pitch`
- `float32 yaw`
- `float32 altitude`
- `string mode`
- `bool armed`
- `builtin_interfaces/Time stamp`

### 10.3 `ControlCmd.msg` 建议字段

- `float32 yaw_rate`
- `float32 vx`
- `float32 vy`
- `float32 vz`
- `bool valid`
- `builtin_interfaces/Time stamp`

### 10.4 `TrackingState.msg` 建议字段

- `bool tracking`
- `bool target_valid`
- `float32 ex`
- `float32 ey`
- `float32 confidence`
- `builtin_interfaces/Time stamp`

## 11. 典型应用场景

### 11.1 目标跟随

这是系统的核心场景。

流程如下：

- YOLO 检测到目标
- 融合层计算目标相对误差
- 控制器输出飞行控制量
- 飞控执行速度和航向控制
- 无人机持续跟踪目标

### 11.2 室内飞行

在室内环境下，无 GPS 情况下也可运行：

- 飞控通过光流、激光、IMU 保持状态估计
- 地面端依然只读取 `/drone/state`
- 整体架构不需要改变

### 11.3 室外飞行

在室外环境下：

- 飞控使用 GPS、IMU、磁罗盘等
- 地面端仍只消费统一状态接口
- 感知与控制结构无需变化

### 11.4 视觉降落

可扩展为：

- 检测降落标志
- 对准降落点
- 飞控负责高度控制与最终降落动作

### 11.5 仿真测试

系统可在以下环境中验证：

- Gazebo
- AirSim
- ArduPilot SITL
- PX4 SITL

建议所有控制链路先在 SITL 完成闭环验证，再进入实飞。

## 12. 延迟结构分析

预估时延如下：

- 图传链路：100 到 200 ms
- YOLO 推理：20 到 60 ms
- ROS2 处理：5 到 10 ms
- 总体端到端：150 到 300 ms

这意味着：

- 系统适合中低速视觉跟踪
- 控制参数必须保守
- 要避免高频大幅控制振荡
- 必须以“稳定可用”优先，而不是“极限灵敏”

## 13. 设计原则与硬约束

这一部分应视为系统的强约束。

### 13.1 明确不做的事情

- 不在 ROS2 正常控制链路中传整路图像
- 不在无人机上跑 YOLO
- 不直接把原始传感器绕过飞控接到 ROS2 做主状态估计

### 13.2 明确必须做到的事情

- 控制链路只传结构化数据
- 控制器独立运行，频率保持在 20 到 50 Hz
- 只处理最新一帧和最新一份状态
- 全链路都必须有超时与有效性检查
- 飞控始终负责底层稳定控制

## 14. 系统优势

该架构的主要优势包括：

- 机载系统简单
- 算法开发效率高
- 地面端算力强，易于使用 GPU
- 调试方便，日志与可视化更容易
- 室内外可统一架构
- 便于后续模块化扩展

## 15. 系统限制

该架构的主要限制包括：

- 图传延迟不可忽视
- 对通信稳定性要求高
- 不适合高速目标追逐
- 单纯图像误差控制下定位精度有限
- 目标遮挡、丢失时性能会明显下降

这些限制必须在控制策略和测试计划里被明确考虑。

## 16. 安全机制与失效保护建议

虽然当前阶段是总策划，但安全设计必须提前纳入。

建议至少具备以下安全机制：

- 目标丢失超时后自动悬停或刹停
- 遥测超时立即停止外部控制
- 感知超时立即停止控制输出
- 支持 RC 或地面站手动接管
- 只有在 GUIDED / OFFBOARD 等允许外部控制模式下才发送控制
- 飞控侧配置高度限制、速度限制和电子围栏

建议任务状态机：

- `IDLE`
- `SEARCH`
- `TRACKING`
- `TARGET_LOST`
- `HOLD`
- `LANDING`
- `FAILSAFE`

## 17. 分阶段开发路线图

为了降低项目风险，建议按阶段推进。

### 第一阶段：系统架构和接口定义

目标：

- 确定 ROS2 包结构
- 确定 Topic 和消息接口
- 确定遥测输入方案
- 确定 YOLO 输出协议

交付物：

- 基于模拟数据的 ROS2 空壳系统

### 第二阶段：视觉感知接入

目标：

- 打通视频输入
- 在地面端运行 YOLO
- 输出稳定目标结果

交付物：

- 实时发布 `/perception/target`

### 第三阶段：飞控遥测接入

目标：

- 通过 MAVLink 连通飞控
- 发布 `/drone/state`
- 检查模式、心跳和链路状态

交付物：

- 使用真机或 SITL 提供稳定状态流

### 第四阶段：融合与控制

目标：

- 实现目标与状态融合
- 实现基础跟踪控制器
- 加入目标丢失处理

交付物：

- 在仿真中稳定输出 `/control/cmd`

### 第五阶段：执行与 SITL 闭环

目标：

- 将控制量真正映射到 MAVLink 指令
- 在 SITL 中验证闭环
- 完成基础参数整定

交付物：

- 仿真中的端到端闭环跟踪能力

### 第六阶段：实飞验证

目标：

- 先进行低风险室内或保护条件下测试
- 限制高度与速度
- 验证安全接管与失效保护

交付物：

- 首次真实环境跟踪测试通过

## 19. 最终定位总结

本项目是一个地面计算式无人机视觉跟踪系统。

它的本质特点是：

- 感知在地面
- 控制决策在地面
- 飞控负责底层执行与稳定
- 通信链路负责闭环连接

其工程意义在于：

在不增加机载复杂度的前提下，获得更强的感知能力、更灵活的控制开发方式，以及更快的系统迭代速度。
